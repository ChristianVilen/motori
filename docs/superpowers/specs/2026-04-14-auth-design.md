# Vuokramoto ‚Äî Auth Design

## Scope

Email/password authentication with email verification. Social login (Google, Meta) and password reset are deferred ‚Äî see **Out of Scope** below.

---

## Architecture

### Libraries

- **BetterAuth** ‚Äî auth framework. Handles sessions, email/password, email verification tokens, cookie management.
- **BetterAuth Kysely adapter** ‚Äî connects BetterAuth to our existing Kysely + PostgreSQL setup.

### Email Abstraction

`src/lib/email.ts` exposes a single `sendEmail({ to, subject, html, text })` function. In development (no `RESEND_API_KEY`), it pretty-prints to the terminal with a formatted box and any URLs clickable. Adding Resend is one uncommented block ‚Äî no other files change.

### Session Access

A `getSession` server function (`src/lib/session.ts`, using TanStack Start's `createServerFn`) reads the current session from request headers via `auth.api.getSession`. Protected routes call this in `beforeLoad`.

## Database

Two Kysely migrations (in order):

1. **BetterAuth tables** ‚Äî generated by `npx better-auth generate`, outputs `user`, `session`, `account`, `verification` tables. `user` is a PostgreSQL reserved word and must be quoted.
2. **Profile table** ‚Äî `profile` keyed on `user_id` (FK ‚Üí user). Presence of a row indicates profile completion.

### Profile table columns

| Column | Type | Notes |
|--------|------|-------|
| `user_id` | text PK | FK ‚Üí user(id), cascade delete |
| `display_name` | text | Required at profile completion |
| `city` | text | Optional |
| `phone` | text | Optional |
| `show_phone` | boolean | Default false ‚Äî gates phone visibility on listings |
| `license_class` | text | A1 / A2 / A ‚Äî optional |
| `language` | text | fi / en ‚Äî default fi |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | |

## Routes

| Route | Type | Notes |
|-------|------|-------|
| `/api/auth/$` | API catch-all | BetterAuth handles all auth endpoints |
| `/auth/login` | Public | Email + password form |
| `/auth/register` | Public | Registration form with password strength bar |
| `/auth/verify-email` | Public | "Check your inbox" holding page |
| `/auth/complete-profile` | Auth-required | Display name, city, phone, license class |
| `/_protected` | Layout | beforeLoad gate: logged in + email verified |

### Auth flow

1. Register ‚Üí BetterAuth creates user (unverified) ‚Üí verification email sent ‚Üí redirect to `/auth/verify-email`
2. User clicks link ‚Üí BetterAuth verifies ‚Üí redirects to `/auth/complete-profile`
3. Profile saved ‚Üí redirect to `/` (or original `?redirect=` target)
4. Subsequent logins: login ‚Üí if profile exists ‚Üí redirect home; if profile missing ‚Üí `/auth/complete-profile`

### Verification gate

Unverified users: can browse all public routes. Blocked from any route under `/_protected` (create listing, dashboard, contact reveal).

Profile completion is not a hard gate on protected routes ‚Äî it's reached via the post-verification redirect. Users can skip it and complete it later from settings (P1).
